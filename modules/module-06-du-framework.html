DU Framework: End-to-End Attended Automation

Introduction to the RPA Framework for Document Understanding

As Document Understanding implementations grew in complexity, UiPath recognized the need for a dedicated RPA framework specifically designed for document processing workflows.

Why a Dedicated DU Framework?

Document Understanding processes have a unique logical flow and particular requirements that differ from standard RPA automations. The existing REFramework (Robotic Enterprise Framework) was not designed for:

  • Long-running workflows with human validation pauses
  • Document-level state management (tracking per-document status)
  • Confidence-based branching logic
  • Integration with Action Center for human-in-the-loop processing
  • Classifier and extractor feedback/retraining workflows

Key Features of the DU Framework

  • ✅ Production-ready: Built-in logging, exception handling, and retry mechanisms
  • ✅ Flexible: Suitable for both attended and unattended (Action Center) implementations
  • ✅ Easy to switch: Toggle between attended and unattended with minimal code changes
  • ✅ Best practices built-in: RPA, DU, Orchestration, and long-running workflow patterns
  • ✅ Scalable: From quick demos to full enterprise implementations
  • ✅ Developer-friendly: Easier development, testing, deployment, debugging

Generic Document Processing Flow

Understanding the complete logical flow of a DU automation helps you build robust, maintainable workflows. Here's the full pipeline:

#StepDescription
1Pre-digitization ProcessingPreprocessing before OCR — image enhancement, grayscale, skew correction, noise removal
2Digitize DocumentRun OCR to convert document to machine-readable text and Document Object Model
3Classify DocumentIdentify document type using configured classifiers
4Classification Business Rule ValidationCheck if classification confidence meets threshold — determine if human review is needed
5Human Classification / Validation (if needed)Route to Action Center for human to review and correct classification
6Pre-extraction ProcessingDocument splitting, page selection, any required preprocessing before extraction
7Data ExtractionRun extractors to pull field values from the classified document
8Human Validation (if needed)Route low-confidence fields to Action Center for human review
9Extractor Feedback (Learning)Send validated corrections back to retrain the model
10Post-extraction ProcessingData formatting, business rules application, calculated fields
11Data ExportSend extracted data to downstream systems (ERP, SAP, database)
12Post-export ProcessingAny cleanup, merging exports, notifications
📌 Note
Not all steps are required for every use case. In practice, some steps will be merged or excluded based on your specific process requirements. Design your flow with the minimum necessary steps to avoid unnecessary complexity.

Specific Challenges in DU Automation

Challenge 1: Document Quality Variability

Real-world documents vary significantly in quality — scanned at different DPIs, rotated, stained, or with handwriting. Solution: add pre-processing steps (image enhancement) and handle OCR failures gracefully.

Challenge 2: Confidence Threshold Management

Setting the right confidence thresholds is critical. Too high → too many human validations. Too low → too many errors pass through. Solution: start conservative, monitor production performance, and tune thresholds over time.

Challenge 3: Long-Running Workflows

When a document needs human validation, the automation must pause and wait — sometimes for hours or days. Solution: use Orchestrator's Long-Running Workflow pattern with Action Center integration.

Challenge 4: Mixed Document Batches

Many real-world processes receive batches containing multiple document types in a single file or folder. Solution: implement document splitting and per-document type routing logic.

Challenge 5: Retraining Without Disruption

Models need regular retraining as new document variations emerge. Solution: implement classifier and extractor feedback loops, and use A/B testing when deploying new model versions.

Solution Architecture

A well-designed DU automation architecture separates concerns into distinct layers:

Recommended Architecture Layers

LayerComponentResponsibility
InputDispatcher ProcessScan input folders, create queue items, deduplication
ProcessingPerformer Process (DU Framework)OCR, classify, extract, validate, export
Human LoopAction CenterHuman validation of low-confidence results
LearningFeedback WorkflowsSend corrections back to retrain models
MonitoringOrchestrator + InsightsTrack SLAs, throughput, accuracy metrics
OutputDestination SystemsERP, database, SharePoint, email
💡 Tip
Attended vs Unattended: The same DU Framework project can be configured for both attended (human triggers, validates in real-time) and unattended (fully automated with Action Center for async validation). The key difference is how the human validation step is triggered.

Custom Success Logic

Real business processes rarely have simple success/failure outcomes. The DU Framework allows you to implement custom success logic based on your business rules.

Examples of Custom Success Logic

  • Conditional routing: If invoice total > $10,000, route to senior approver
  • Multi-field validation: Accept extraction only if at least 5 out of 7 required fields are populated
  • Cross-field validation: Line item total must match the invoice total (±2% tolerance)
  • Duplicate detection: Check if invoice number already exists in the system before processing
  • Date validation: Reject invoices older than 90 days

Implement custom success logic after the Data Extraction Scope and before the Export step. Use an If/Switch activity to branch based on your business rules.

Error Handling in DU Workflows

Error Handling Strategy

Error TypeHandling Approach
OCR/Digitization failureLog error, move to failed folder, continue with next document
Classification failureRoute to human classification or mark as "unclassified"
Extraction error (technical)Retry logic (max 3 attempts), then escalate to human
Low confidence (not an error)Route to Action Center for validation — not a failure
Downstream system errorQueue document for retry, alert operations team
Timeout (human validation)Escalate after SLA breach, notify supervisor

Best Practices for Error Handling

  • Always use Try/Catch blocks around extraction activities
  • Log errors with document name, error message, and timestamp
  • Never silently swallow exceptions — always log at minimum
  • Separate technical errors from business exceptions
  • Implement dead-letter queues for persistently failing documents
  • Use Orchestrator queue item statuses to track document processing state
