What is UiPath Document Understanding?

⏱ 30 minutes📖 Theory🎯 Beginner–Intermediate

🎯 Learning Objectives

  • Describe the UiPath Document Understanding overview and product architecture
  • Identify different document types and when to use each processing methodology
  • Explain taxonomy — document types, classification, and field definitions
  • Describe the end-to-end DU processing pipeline: Digitize → Classify → Extract → Validate → Export
  • Understand confidence scores and their role in human-in-the-loop validation

UiPath Document Understanding Overview

UiPath Document Understanding (DU) is a product within the IXP platform that enables robots to read, understand, and extract data from business documents — just like a human would — but at scale and speed.

Key Capabilities of Document Understanding

  • Digitization: Convert physical or image documents to machine-readable text using OCR engines
  • Classification: Automatically identify what type of document it is (invoice, purchase order, tax form)
  • Data Extraction: Pull specific fields and values from the document
  • Human Validation: Route low-confidence results to humans for review via Action Center
  • Continuous Learning: Use human corrections to retrain and improve models over time

Document Processing Methodologies

UiPath DU supports multiple extraction methodologies — each suited to a different document complexity level:

🔧 Rule-Based (Template-Based) Extraction

Relies on fixed rules, templates, or regular expressions. Best for structured documents with consistent layouts.

  • RegEx Extractor: Pattern matching for fields like dates, phone numbers, IDs
  • Form Extractor: Uses anchor-based approach to locate fields relative to known text
  • Forms AI: Combines ML with form understanding for both structured and some semi-structured

Advantages: High accuracy for known documents.

🤖 Model-Based (ML) Extraction

Uses Machine Learning models trained on labeled document examples. Best for semi-structured documents.

  • Out-of-the-Box ML Extractors: Pre-trained models for invoices, receipts — ready to use immediately
  • Modern Projects (Active Learning): New experience combining annotation, training, and deployment in one UI

Key Insight: ML models learn from characteristics of data — position on the page, correlation of text, and the value itself. They are NOT a database of layouts — they truly understand the document.

⚡ Hybrid Extraction

Combines rule-based and model-based approaches. Best for documents that mix structured sections (tables, checkboxes) with semi-structured content.

⚠️ Important
Common Misconception: Many believe ML extractors are "a smart big database of layouts." In reality, ML models learn from characteristics like position, text correlation, and values — making them flexible across new document layouts they've never seen.

Taxonomy in Document Understanding

Taxonomy is the metadata collection that guides the Document Understanding framework throughout its operations. It acts as a single source of truth — defining what documents to expect and exactly what data to extract from them.

Hierarchical Structure

Taxonomy follows a three-level hierarchy:

  1. Taxonomy — The complete metadata collection for the project
  2. Document Types — Logical document categories (e.g., Invoices, Medical Records, Contracts), each with a unique ID. These enable both classification and extraction.
  3. Fields — Individual data points to be extracted from a specific document type (e.g., Invoice Number, Due Date, Total Amount)

Additional Taxonomy Features

  • Exception Handling: Predefined "Report as Exception" reasons appear in the Validation Station, allowing users to flag documents with specific issues
  • Language Support: Taxonomies can specify supported languages with language codes, and include an "Undetermined Language" (code und) for edge cases
  • Storage: Taxonomies are stored as taxonomy.json files in the DocumentProcessing folder within your UiPath Studio project
  • Taxonomy Manager: Managed through the built-in Taxonomy Manager wizard in Studio and loaded at runtime via the Load Taxonomy activity
💡 Tip
Best Practice: Define your taxonomy before building any model. A well-designed taxonomy saves significant rework later. Work with a Subject Matter Expert (SME) who understands both the business process and all edge cases in the documents.

Building a Good Taxonomy

  1. Identify all document types in your process
  2. For each document type, list all data points that need to be extracted
  3. Assign the correct field type to each field (Number, Date, Table, etc.)
  4. Group related fields using FieldGroup for hierarchical data (e.g., line items)
  5. Define exception reasons for documents that cannot be processed
  6. Test with sample documents before investing in large-scale labelling

Classification, Extraction & Validation

Document Classification

Classification is the process of identifying what type of document is being processed. UiPath DU supports:

  • Keyword-Based Classifier: Looks for specific keywords to identify document type
  • Intelligent Keyword Classifier: Enhanced keyword classifier with better handling of variations
  • ML Classifier: Machine learning-based classification — more robust across varied document layouts
  • Helix Classifier (Modern): Next-gen classifier available in Modern projects using AI Center

Data Extraction

Extraction retrieves specific field values from the classified document. The Data Extraction Scope activity in Studio orchestrates the extraction process using the configured extractors.

Validation & Human-in-the-Loop

Not all extractions will be 100% accurate. UiPath DU uses confidence scores to determine when human review is needed.

How Confidence Scores Work

  • Each extracted field gets a confidence score from 0.0 to 1.0 (0% to 100%)
  • You configure threshold values — e.g., "automatically accept if confidence > 80%"
  • Fields below threshold are sent to UiPath Action Center for human validation
  • Human corrections are fed back to retrain the model (continuous learning)
✅ Success
Human-in-the-Loop Best Practice: Start with a lower automatic acceptance threshold (e.g., 90%) to catch more errors early. As the model improves through retraining, gradually raise the threshold to reduce human workload.

End-to-End Processing Pipeline

A complete Document Understanding automation follows this logical flow:

1
Pre-digitization Processing
Any processing before OCR — grayscale conversion, skew correction, noise removal
2
Digitize Document
Convert document to machine-readable text using the configured OCR engine
3
Classify Document
Identify the document type using a classifier
4
Classification Validation
Check if classification confidence meets threshold; send to human if not
5
Pre-extraction Processing
Document splitting (multi-document PDFs), page selection
6
Data Extraction
Run configured extractors to pull field values
7
Human Validation
Route low-confidence extractions to Action Center for review
8
Extractor Feedback (Learning)
Use corrections to retrain the model
9
Post-extraction Processing
Data formatting, business rule application
10
Data Export
Send extracted data to downstream systems (ERP, database, API)
📌 Note
Not all steps are required for every use case. Simple processes may skip classification, human validation, or feedback steps. Design your flow based on your specific business requirements.
